# `mcp-tool-schema` reference: implementation reconciliation against the current MCP server

This reference grounds the implementation plan for the accepted `mcp-tool-schema` ADR by reconciling the approved nine-tool design against the code that exists today. It settles three questions the ADR left to implementation: how batch editing supports reconciliation workflows, what current code is rebuilt versus discarded versus reused, and where the boundary sits between in-process dispatch and subprocessing the CLI. The four ADR-amending refinements it produced are already folded into the ADR body; everything else here is implementation grounding within the ADR's existing decisions.

## Synthesis

All three questions resolve against one observation from the code: the shared-core layer the ADR's "route through owning verb logic" principle presumes already exists for four of the five mutation domains - `vaultcore.hydration.create_vault_doc` owns creation, `plan.commands.step_ops` plus `plan.parser` own plan structure, `vaultcore.orientation` owns status rollups, `vaultcore.checks` owns validation - and is missing for exactly one: the body-edit pipeline, which lives as Typer-coupled helpers in `src/vaultspec_core/cli/edit_cmd.py` (`_execute_edit` and its satellites). The current MCP `create` is the only true drift casualty: it reimplements filename generation, hydration, and validation in parallel to `create_vault_doc` and must be rebuilt onto it. That settles the boundary: hot tools converge in-process onto `vaultcore`/`plan` core functions (cheap, mostly done), the edit engine gets one bounded extraction, and `invoke` becomes a subprocess of the installed `vaultspec-core` binary with `--json` - the only disposition that covers 124 Typer-coupled verbs without unbounded plumbing. Batch semantics adopt the CLI's already-shipped optimistic-concurrency contract (`--expected-blob-hash`, `git_blob_oid` in `vaultcore/blob_hash.py`) and its canonical status vocabulary, so the MCP surface and the CLI speak one result language.

## Q-A. Batch editing as a first-class reconciliation surface

Decision: adopt the CLI's blob-hash optimistic-concurrency contract per edit operation, return the post-write hash per item, and unify create/edit under one per-item result envelope using the CLI's canonical status vocabulary. Rejected alternative: no concurrency guard (last-write-wins), because the refute/confirm loop is precisely the read-modify-write race the CLI already solved for the dashboard engine via `vault set-body --expected-blob-hash`; omitting it on the MCP surface would make the MCP path weaker than the CLI path it wraps.

Specifics, grounded in `edit_cmd.py`:

- Optimistic concurrency. Each `edit` operation carries an optional `expected_blob_hash`; enforcement reuses `_enforce_blob_hash` semantics (git blob OID of pre-write on-disk bytes, byte-compatible with the dashboard engine). Every successful per-item result returns the post-write `blob_hash` - the CLI envelope already does exactly this (`_execute_edit` emits `blob_hash` on `updated`/`unchanged`), so the agent chains edit N+1 from edit N's result without a re-read.
- Read-side composition. `find` in document-search mode returns `blob_hash` per result entry (cheap at `limit=20`; bytes are already read when `body=true`), so the canonical reconciliation loop is find-with-body, then edit-with-expected-hash, then chain from returned hashes. `status` stays orientation-level and returns no hashes; feature-listing mode of `find` returns none.
- Intra-batch same-document rule. Operations apply strictly sequentially; `expected_blob_hash` is validated against on-disk bytes at the moment the item applies. An agent stacking multiple ops on one document in a single batch sets the hash on the first op only and omits it on subsequent ops. Rejected: server-side automatic hash rebinding within a batch, which would mask genuine external conflicts behind magic.
- Section addressing. `append_section` / `replace_section` / `set_body` is the right granularity: reconciliation edits are overwhelmingly "update this section's prose," and section ops avoid resending whole bodies. Sections are addressed by exact heading-line text, first match; a miss is a per-item failure (`section_not_found`), never a fuzzy match. All three ops compose the new full body and flow through the extracted set-body engine, so the `modified:` stamp refresh, pre-write conformance checks (`_validate_proposed` pattern: frontmatter, links, body-links over an in-memory snapshot), refuse-on-error, and hash guard apply uniformly. Note `append_section` is retry-hostile in isolation; the hash contract neutralizes this - a retry with a stale hash conflicts instead of duplicating content.
- Failure model and shared envelope. Per-item results, no all-or-nothing. Item shape is shared between `create` and `edit`: index, target, `status` from the CLI's canonical vocabulary (`created` / `updated` / `unchanged` / `failed`), `path`, `blob_hash` on success, structured `error` (with `conflict` / `refused` / `checks` payloads mirroring `_EditError` data) on failure, plus `warnings`. The whole-call `structuredContent` carries an aggregate `status` of `ok` / `mixed` / `failed`, matching the CLI's sync-envelope aggregate. Retry protocol: resubmit only items whose `status` is `failed`, with refreshed hashes for `conflict` items. Batch-create's intra-batch lifecycle validation (each item validated against vault state including earlier same-batch items) reports through this same shape - a dependency violation is an item-level `failed` with the `validate_feature_dependencies` diagnostic, and later items still apply.

## Q-B. Disposition of the current implementation

Decision: converge all MCP mutation onto the shared core layer that already exists in `vaultcore` and `plan` - functions below Typer and above the vault filesystem that take `root_dir`/paths, return data or typed results, and raise typed errors - with exactly one extraction (the edit pipeline) and one small normalizer consolidation. Rejected alternative: keeping the MCP tools as a parallel implementation, because the drift is already observable (MCP `create` builds filenames by hand while `create_vault_doc` owns filename generation; MCP `create` skips index regeneration; two divergent kebab-case validators), and every future CLI fix would need mirroring by hand.

## Q-C. The invoke boundary

Decision: hot tools dispatch in-process to the shared core functions above with rich typed schemas; `invoke` subprocesses the installed `vaultspec-core` binary as an argv-list (never shell) invocation, appending `--json` where the inventory says the verb supports it and wrapping raw stdout as text content otherwise, with verb-path and argument validation against the parsed `cli.md` inventory before spawn. The boundary is exactly "hot = in-process core, long tail = subprocess CLI."

Why Option 2 (subprocess) for the gateway, against the alternatives:

- Option 1 (in-process for `invoke` too) requires the 100-plus long-tail verbs to be callable outside Typer. They are not: `cmd_*` functions render envelopes, print Rich output, and raise `typer.Exit`; global path state is process-wide via `init_paths`. Making them all callable is the unbounded version of the edit-engine extraction, for verbs the analytics show are invoked rarely or never. In-process CliRunner-style capture inside a long-running async server is worse - shared stdout, global context mutation across concurrent requests, and a crashing verb takes the server with it.
- Option 2 gives uniform coverage of the whole 124-verb surface with zero reimplementation and a single source of truth that is literally the installed binary - the same integrity argument the ADR's Q7 makes for the marker-parsed catalog, now extended to execution: the catalog cannot advertise fiction, and the executor cannot behave differently from the binary. Process isolation is free; a fresh process gets clean path context per call. Spawn latency (roughly 100-300ms plus Python startup) is acceptable precisely because the long tail is low-frequency by definition - the 3.5k-per-month operations never pay it, since they are in-process tools.
- Option 3 (no gateway, Bash plus CLI) is rejected on the analytics: agents already do this and it is the 7.9%-miss, retry-after-help pattern the MCP exists to replace. `discover` plus `invoke` keeps schema-validated arguments, one consistent tool call, and host-visible destructive annotation - none of which raw Bash provides.
- Interaction with Q-B: the subprocess boundary also removes the drift pressure. If `invoke` were in-process, every long-tail verb would eventually demand its own core extraction; subprocessing caps the extraction obligation at exactly the hot-tool set.
- Interaction with the dependency fork: this strengthens the stay-on-`mcp>=1.26.0` decision - the gateway is now a catalog parser, an argv builder, and a JSON parser, needing nothing from standalone `fastmcp`.

Security note for the plan: `invoke` builds an argv list from the validated verb path plus schema-validated arguments, injects `--target` from the server's resolved `root_dir`, and never routes through a shell; the catalog validation is the injection guard, and the static denylist (`uninstall`, MCP registry mutation, index hand-authoring) is enforced at both `discover` and `invoke`.

## ADR amendments (folded into the accepted ADR body)

These refine three-to-four sentences rather than reverse any decision, so they were folded into the ADR in place rather than superseding it:

- Implementation `invoke` description now reads: after validating the verb path and arguments against the parsed inventory and denylist, `invoke` executes the installed `vaultspec-core` binary as an argv-list subprocess (never a shell), injecting `--target` and passing `--json` where the verb supports structured output.
- Q3 rationale: "`invoke` executes the installed binary, which is the owning verb logic" (was "dispatches directly to the owning verb logic").
- Q5 `edit` shape gained the `expected_blob_hash` optimistic-concurrency guard and post-write `blob_hash` return.
- Implementation `find` gained per-document `blob_hash` for read-then-edit chaining.

## Plan implications

- Phase-shaping extraction first: move the edit pipeline helpers out of `cli/edit_cmd.py` into a `vaultcore` edit-engine module returning typed results, and re-point `cmd_set_body` / `cmd_set_frontmatter` / `cmd_edit` at it, before any MCP `edit` work; this is the only blocking refactor and it de-risks everything downstream. Include the small kebab-case normalizer consolidation here.
- The `create` rebuild is a deletion plus a thin batch loop over `create_vault_doc` and `generate_feature_index`; no new creation logic may be authored in `mcp_server/`.
- `plan_progress` / `plan_edit` / `status` / `check` are thin-wrapper phases over `plan.commands`, `vaultcore.orientation`, and `vaultcore.checks`; plan resolution by feature or stem needs a small shared resolver (feature to unique plan document, ambiguity is an error).
- The gateway is two deliverables: a catalog module parsing the `vaultspec:generated` markers in `.vaultspec/reference/cli.md` with the static denylist, and a subprocess executor with argv construction, `--json` envelope parsing, timeout, and stderr capture into the error payload.
- Cross-cutting closeout phase: `isError` migration, `outputSchema` on all nine tools, corrected annotations per ADR Q6, `_isolated_context` on every handler, and the unified batch result envelope defined in Q-A.
- Testing: WorkspaceFactory-based, real filesystem, no mocks - the blob-hash conflict path, intra-batch same-document sequencing, partial-failure envelopes, and the subprocess executor against the real binary all need direct tests.
- Docs dependency: gateway usefulness is bounded by description quality inside the generated inventory; a plan step should audit the hot-verb help strings in the CLI reference, since they become the `discover` payload verbatim.

Key paths named for the writer: `src/vaultspec_core/mcp_server/app.py`, `src/vaultspec_core/mcp_server/vault_tools.py`, `src/vaultspec_core/cli/edit_cmd.py`, `src/vaultspec_core/cli/vault_cmd.py` (`cmd_add`, `generate_feature_index` usage), `src/vaultspec_core/cli/plan_cmd.py`, `src/vaultspec_core/cli/status_cmd.py`, `src/vaultspec_core/vaultcore/hydration.py` (`create_vault_doc`), `src/vaultspec_core/vaultcore/blob_hash.py`, `src/vaultspec_core/vaultcore/orientation.py`, `.vaultspec/reference/cli.md` (the `vaultspec:generated` markers).
