# `curator-reframe` research: `curator reframe to ADR architecture reconciliation`

This research grounds the reframe of the `vaultspec-curate` skill and its
`vaultspec-docs-curator` agent persona. Both were authored when the curator was the
only mechanism for keeping `.vault/` tidy, so they read as a mechanical schema janitor:
frontmatter shapes, wiki-link quoting, filename patterns, tag pairs, template
compliance. The `vaultspec-core` CLI has since absorbed nearly all of that mechanical
work behind deterministic verbs. The persona is therefore redundant where it is
mechanical and absent where it is now most valuable: keeping the architecture decision
corpus and the codebase a single, curated, non-contradictory set of decisions. The
intent of this reframe is to retarget the curator as an ADR architecture semantic
reconciliation persona that cedes mechanical hygiene to the CLI and spends its judgment
on decision-vs-decision and decision-vs-code conflicts. Edit target is
`src/vaultspec_core/builtins`; the rendered `.vaultspec/` and provider copies are
regenerated by sync and are out of scope. The names are kept.

## Findings

### Current curator versus the target

The shipped skill and persona operate in an Audit then Delegate then Verify loop whose
entire vocabulary is structural: Class A frontmatter violations, Class B tag hygiene,
Class C reference integrity, Class D filename and path integrity. Every one of those
classes now maps onto a CLI check (see below), so the persona spends its reasoning
re-deriving rules the CLI already enforces, and delegates the fixes it could ask the CLI
to apply directly. Nothing in the current persona reasons about whether an `accepted`
decision is actually reflected in the code, whether two ADRs contradict each other, or
whether a superseded decision still governs the implementation. That semantic layer is
the target.

### The mechanical harness is now CLI-owned

The `vaultspec-core` CLI exposes a complete mechanical curation surface that supersedes
the persona's hand work. The relevant verbs:

- `vaultspec-core vault check` is a suite of named checks: `frontmatter`, `links`,
  `body-links`, `dangling`, `orphans`, `references`, `schema` (enforces ADRs must
  reference research and plans must reference ADRs), `structure`, `encoding`,
  `placeholders`, `annotations`, `markdown`, `modified-stamp`, `features`,
  `rename-integrity`, `feature-rename-integrity`, plus `all`. Most accept `--fix` for
  safe auto-corrections, `--feature` scoping, and `--json`. This is the entire Class A
  through Class D surface of the old persona, made deterministic.
- `vaultspec-core status [TARGET]` is read-only orientation: a vault-wide rollup, or a
  grounding trace for a plan or feature handle. `--json` for machine reads.
- `vaultspec-core vault list [adr] --json` enumerates documents with path, name,
  doc_type, feature, date, and tags. Note: it does not surface ADR status, because
  status lives in the document body, not the frontmatter (see status conventions below).
- `vaultspec-core vault stats --invalid --orphaned --json` reports health metrics.
- `vaultspec-core vault graph` renders the decision topology: `--feature` scoping,
  `--metrics`, `--json` node-link export, `--node <stem> --depth N` for a local ego
  neighbourhood, `--ref <branch|sha>` to read the corpus from a git ref without a
  checkout, and a derived relatedness edge set. The `supersedes` / `superseded_by` and
  `related:` edges are first-class graph data.
- `vaultspec-core vault adr supersede OLD --by NEW` is the one ADR-lifecycle mutation
  primitive. The implementation in `src/vaultspec_core/core/adr.py` writes
  `superseded_by: '<new>'` on the old ADR, appends the old stem to the new ADR's
  `supersedes:` list, rewrites the old ADR H1 status token to `superseded`, and refreshes
  the `modified` stamp on both. It supports `--dry-run` and `--json`.
- `vaultspec-core vault link list|add|remove` is edge CRUD on the `related:` frontmatter
  graph.
- `vaultspec-core vault set-body`, `set-frontmatter`, `edit`, and `rename` are the
  canonical surgical mutators. They keep the frontmatter contract and the `modified`
  stamp correct, so a curator that rephrases or amends ADR prose should prefer
  `vault set-body` or `vault edit` over a raw file edit.
- `vaultspec-core doctor` diagnoses overall workspace and vault health.

The design consequence is direct: the reframed curator never hand-fixes frontmatter,
links, names, or templates. It runs `vault check all --fix` as a precondition so the
corpus is structurally correct before any semantic work begins, then spends its effort
on what no check can decide.

### vaultspec-rag ADR-scoped semantic search

The semantic engine for reconciliation is `vaultspec-rag search`. The affordances that
matter here:

- `vaultspec-rag search "<intent>" --type vault --doc-type adr` is the directed ADR
  recall: decision discovery by meaning, scoped to the ADR corpus. It also takes
  `--feature`, `--tag`, `--date`, `--scores`, and `--json`.
- `vaultspec-rag search "<concept and domain nouns>" --type code` finds the
  implementation sites a decision should govern, with `--language`, `--path`,
  `--include-path` / `--exclude-path` globs, `--function-name`, `--class-name`,
  `--prefer production|tests|documentation`, and `--structure` filters.
- The same searches are available over MCP as `search_vault` and `search_codebase`.

Operational gotcha, confirmed live in this worktree: the rag service was ready (server
backend, CUDA, all models present) but reported zero indexed vault documents and zero
indexed code sections. The file watcher reindexes on change, but a freshly checked-out
worktree with no edits is never triggered, so the index is empty until something writes.
The reconciliation workflow therefore cannot assume a populated index. The skill must
treat indexing as an explicit precondition: check `get_index_status`, and if empty,
trigger `reindex_vault` and `reindex_codebase` (or the `vaultspec-rag index` CLI path)
before relying on semantic recall. Where rag is unavailable, the discovery rule's
fallback of CLI discovery verbs plus grep carries the same locate-read-confirm sequence.

### ADR corpus status and supersession reality

The corpus is 82 ADRs. Status is encoded in the document body H1 as a parenthesised,
backtick-quoted token after the title (for example a status value of `accepted`). The
live distribution exposes real drift:

- Roughly 68 `accepted` (split between backtick-quoted and bare tokens), 5 `proposed`,
  zero `rejected`, and zero `deprecated` or `superseded`.
- 10 ADRs use a legacy `## Status` section with a bare value such as `Accepted` rather
  than the H1 token, and one encodes status inside a table. So at least three
  incompatible status encodings coexist.
- The status vocabulary itself is contradictory across sources. The ADR template comment
  declares the set `proposed | accepted | rejected | deprecated`. The `adr supersede`
  tool writes `superseded`, a fifth value not in the template set. There is no status
  enum anywhere in the code: `src/vaultspec_core/core/enums.py` has no ADR status type,
  and no `vault check` validates status. Status is unvalidated free text.

Two consequences for the reframe. First, the supersede H1 rewrite uses a regex that only
matches the H1-inline form, so the 10 legacy `## Status` ADRs would receive
`superseded_by` frontmatter while their visible body status stays stale. A curator must
detect that frontmatter-versus-body divergence. Second, the absence of any mechanical
status validator and the template-versus-tooling taxonomy split are genuine
corpus-definition gaps. These are not deferrable: the canonical status set belongs in the
core library, and this feature encodes it (see Criticality below).

### The reconciliation workflow (the HOW)

The validated discovery sequence (locate by meaning, read the epicenter whole, confirm
with grep) maps onto a locate-read-confirm-action reconciliation loop:

- Precondition. Ensure rag vault and code indexes are populated. Run
  `vault check all --fix` so the corpus is structurally clean before semantic work.
- Enumerate and parse. `vault list adr --json` for the inventory; read each H1 (and the
  legacy `## Status` section) to extract the declared status the listing cannot give;
  pull the supersession and relatedness edges from `vault graph --json`.
- Reconcile decision-versus-decision. Use `rag search --type vault --doc-type adr` to
  find ADRs covering the same concept, then read them whole to judge whether they agree,
  duplicate, or contradict. Detect declared-but-unpropagated supersession (frontmatter
  set, body status stale), status-taxonomy violations, and orphaned or stranded
  decisions via the graph.
- Reconcile decision-versus-code. For each `accepted` ADR, `rag search --type code` on
  the decision's domain nouns, read the epicenter file whole, and confirm the decision is
  actually implemented; grep to confirm exact symbols. For superseded or deprecated
  decisions, check that the old approach no longer dominates the code.
- Action. Propagate status mechanically with `vault adr supersede`. Rephrase or amend
  conflicting ADR wording through `vault set-body` or `vault edit`. Surface gaps,
  conflicts, and errors that need human judgment in an audit report scaffolded with
  `vault add audit`.
- Verify. Re-run `vault check all` and re-scan the touched ADRs; loop until clean; persist
  the audit.

### Agent Skills schema compliance

The Claude Code Agent Skills standard governs the rewrite. A skill is a directory with a
required `SKILL.md` whose YAML frontmatter fields are all optional but `description` is
recommended; the combined `description` plus `when_to_use` text is capped at 1,536
characters in the skill listing, so the key use case must come first. Other supported
fields include `name`, `allowed-tools`, `disable-model-invocation`, `user-invocable`,
`model`, `effort`, `context: fork`, `agent`, and `paths`. Guidance: keep `SKILL.md`
under 500 lines and concise, because once invoked its content stays in context for the
session; move detailed reference material into sibling files and point to them from
`SKILL.md` so they load only when needed (progressive disclosure). This pattern is
already used by a vaultspec builtin: the documentation skill ships `references/` and
`agents/` subdirectories, confirming the build and sync pipeline carries skill-bundled
resources. The reframed curator can therefore keep a lean `SKILL.md` and grow a
`references/` folder (for example, the ADR status taxonomy and a reconciliation
playbook) without violating the schema or the build. The agent persona file is separate
and keeps its own frontmatter (`description`, `tier`, `mode`, `tools`).

### Criticality: the status taxonomy is unencoded in the core library

The canonical ADR status set is not defined anywhere in the code. `DocumentMetadata` in
`src/vaultspec_core/vaultcore/models.py` models `supersedes`, `superseded_by`,
`derived_from`, `promoted_to`, and `archived`, but has no `status` field. There is no
status enum in `src/vaultspec_core/core/enums.py`. The only code that touches status is
the `adr_supersede` rewrite in `src/vaultspec_core/core/adr.py`, which hard-codes the
literal `superseded` against an H1-inline regex. The taxonomy therefore exists only as
prose: a template comment that lists `proposed | accepted | rejected | deprecated`, which
the tool then contradicts by writing `superseded`. This is a defect, not an open
question: a semantic-reconciliation curator cannot enforce a taxonomy the library does
not define. This feature resolves it by encoding the canonical status set in the core
library as the single source of truth, with the template, the supersede tool, and the
new validator all deriving from it.

The check surface for the validator is established. The suite lives in
`src/vaultspec_core/vaultcore/checks/`, with the result contract (`CheckResult`,
`CheckDiagnostic`, `Severity`, `VaultSnapshot`) in `_base.py`, each checker re-exported
from `__init__.py` and aggregated by `run_all_checks`, then surfaced under the
`vault check` CLI group. A new `adr-status` checker (canonical encodings, H1-versus-body
divergence, frontmatter-versus-body supersession drift) follows that existing shape.

### Resolved scope decisions

- Taxonomy: encode the canonical status set in the core library as the source of truth;
  reconcile the template and the supersede tool to it. Settled in the ADR from evidence,
  not by preference.
- Autonomy: the curator applies mechanically-safe actions directly (status propagation
  via `vault adr supersede`, format and stamp normalization) and surfaces wording and
  contradiction calls in the audit for human approval before amending.
- Decision-versus-code drift: ADRs drive codebase rollout, so the curator never retrofits
  an ADR to match code automatically. Reporting drift is advisory by default. Amending an
  ADR to reflect existing code (the legitimate ADR-from-codebase retrofit for projects
  adopting the pipeline late) is a valid operation the curator offers but executes only on
  explicit human request, never on its own.
- Mechanical status validator: in scope for this feature, built on the check surface
  above.

### Scope note

This is no longer a prose-only change. The feature now spans the skill and agent builtin
rewrite plus core-library work (the status enum, model and template reconciliation, and
the `adr-status` validator). It warrants an ADR that records the taxonomy decision and the
validator design, and a tiered plan covering both the code and the prose.
