# `install-parity` adr: `companion provisioning parity and the three-placement mode model` | (**status:** `accepted`)

## Problem Statement

`install-mode` (`2026-07-13-install-mode-adr`) gave vaultspec-core a persisted tool-versus-dependency placement axis, but vaultspec-rag, its companion project and a core-provisioned workspace in its own right, has no `--mode` flag, no committed placement declaration, and a static MCP definition that always renders the dependency shape (`2026-07-14-install-parity-research`). Consumers who run core in one placement and rag in another today get silent divergence, not a coherent choice. Separately, the two-mode model itself is incomplete: the default `dev` dependency group renders byte-identically to a full project dependency under `uv sync`/`uv run`, yet core's schema cannot express "dev-scoped, will not leak downstream" versus "runtime dependency, will leak downstream" as distinct declared states. This record amends `install-mode`'s Q1 persistence schema and Q5 flag vocabulary to add that third state, and extends the resulting three-mode model to rag so both CLIs expose the same choice over the same persisted shape.

## Considerations

- Rag's enrollment pipe already reuses core's `seed_builtins`/`sync_provider` byte-for-byte; only rag's mode declaration, MCP rendering, and doctor parity are missing, not the underlying provisioning path (`2026-07-14-install-parity-research`, parity table).
- PEP 621 project dependencies leak into built distributions; PEP 735 dependency groups do not. The default `dev` group renders identically to a project dependency under `uv run`/`uv sync` with no flags, so a third mode scoped to that group is bookkeeping and a sharper doctor label, not a rendering change (`2026-07-14-install-parity-research`, three-placement question).
- Named (non-default) groups are a materially different problem: they need `--group <name>` threaded through every rendered `uv run` call and rag's own torch flow documents `[tool.uv.sources]` pins going silently inert on group-placed dependencies until the group is enabled. Supporting them would persist a group name, not just an enum member, and is out of scope here.
- Ecosystem precedent (pytest, ruff, mypy, prek) never refuses runtime-dependency placement for dev tooling; guidance only, no hard refusal (`2026-07-14-install-parity-research`).
- Companion coupling is a real, likely common case: rag is torch/GPU-heavy, so core-as-dependency with rag-as-tool (or vice versa) is a legitimate simultaneous configuration the current single-key `workspace.json` schema cannot express (`2026-07-14-install-parity-research`).
- This decision touches the same three coupling points `install-mode` already identified (MCP definition, hook/launch rendering, doctor), now duplicated across two repositories; `install-mode`'s Rationale established that all coupling points must move together or drift reappears, and that principle carries over unchanged to the companion surface.

## Considered options

**D1 - third mode.** Chosen: add `InstallMode.DEV`, scoped only to the default `dev` group, with `--mode tool|dependency|dev` in both CLIs; `DEV` renders byte-identically to `DEPENDENCY` and differs only in declared bookkeeping and doctor labeling. Rejected: binary mode with a detection-only "leak detail" flag, because a non-persisted detail cannot survive `doctor` or a fresh clone reading the workspace's own declaration. Rejected: named-group support, because it persists a group name and a group-aware render path, a materially larger surface than this decision's scope.

**D2 - schema shape.** Chosen: `workspace.json` becomes a per-package map as the primary shape (`{"packages": {<name>: {"mode": ..., <floor field>: ...}}}`), with a `schema_version` bump and a migration path that reads the legacy single-key shape. Rejected: single mode plus a per-package `overrides` key, because the research found companion-mode divergence (core-as-dependency, rag-as-tool or the reverse) to be a likely common case, not an edge case an overrides map should carry the connotation of being. Rejected: per-package separate files, because it fragments a single committed declaration into N files with no shared read/write path and multiplies the migration surface instead of collapsing it.

**D3 - runtime-dependency placement enforcement.** Chosen: warn-only when a package's declared mode is `dependency` (full leak) rather than `dev`/`tool`; never refuse the placement. Rejected: hard refusal, which no dev-tool in the ecosystem does and which core's own self-hosting (dependency mode by explicit choice, per `install-mode`) would itself trip. Rejected: silence, which reintroduces the opaque-failure-later pattern `install-mode`'s Problem Statement already closed once.

**D4 - rag's adoption surface.** Chosen: rag adopts the full analogue surface, an install-time `--mode` flag, a tokenized mode-neutral MCP builtin definition substituted by core's existing renderer, shared `workspace.json` read/write via core's `workspace_mode` module (rag already depends on `vaultspec-core`), doctor mode-and-floor rows, and upgrade-time mode inference; rag's `--local-only` backend marker is left untouched as an orthogonal per-host storage choice, not a placement mode. Rejected: a rag-owned separate declaration file, which reintroduces the exact committed-versus-per-machine inversion the research flagged (rag currently has only a per-host marker with no committed counterpart) instead of fixing it. Rejected: rag reimplementing its own mode machinery, which would fork the single-comparator discipline `install-mode` established and the sibling drift that decision existed to prevent.

## Constraints

- Amends, does not supersede, `2026-07-13-install-mode-adr`: that record's Q2/Q3/Q4/Q6 mechanism choices (uvx module invocation, hook entry shape, floor-plus-provision-stamp layering, upgrade inference order) stand unchanged; only its Q1 schema shape and Q5 flag vocabulary are extended with a third state.
- Release ordering is a hard dependency: rag's `--mode` flow calls into core's `workspace_mode` resolution and schema-v2 read/write, so core must ship the `DEV` member and schema v2 first; rag's own `pyproject.toml` floors on that core version (rag already declares `vaultspec-core>=0.1.27` as a direct dependency).
- Schema v2 migration risk lands on a freshly-shipped v1: `install-mode` has not yet had a release cycle to accumulate real single-key `workspace.json` files, so the legacy-read compatibility path is exercised mostly by a narrow window of adopters, but must still be correct and tested, not assumed unreachable.
- Must not introduce a second content or parity comparator between core's and rag's renderers; each repository's renderer/doctor continues to consult its own repository's entry in the shared per-package map, following the "one comparator, several views" shape `install-mode` established.
- Both repositories' test suites must exercise all three modes for both packages' entries, including the cross-configuration case (one package dependency-mode, the other tool- or dev-mode) that motivated D2.

## Implementation

Core gains the `DEV` enum member alongside `TOOL` and `DEPENDENCY`; every renderer that branches on mode treats `DEV` as an alias of `DEPENDENCY` for rendering purposes (uvx-versus-module launch shape, hook entry form) while treating it as a distinct value for doctor labeling and declared-state bookkeeping. `WorkspaceDeclaration` moves from its shipped single-key schema to a `schema_version`-bumped per-package map (`{"packages": {<name>: {"mode": ..., "minimum_vaultspec_version"-equivalent floor}}}`); the read path recognizes the legacy single-key shape and folds it into the new map keyed to the current package on read, writing the migrated v2 shape back out, so no separate one-time migration command is needed. Detection gains a group-aware branch: presence of a package in `[project.dependencies]` is dependency-mode evidence as before, presence only in the default `dev` group is new dev-mode evidence, and named groups continue to fall outside detection's scope. Renderers and the doctor's diagnosis collectors are keyed by the *package's own* declared mode read from its own entry in the shared map: core's renderer and doctor consult core's entry, rag's renderer and doctor consult rag's entry, so a mixed configuration (core dependency-mode, rag tool-mode, or the reverse) renders and diagnoses each package independently without a shared branch.

Rag's `_install.py` gains a `--mode tool|dependency|dev` flag threaded into the same resolution precedence order Q5 of `install-mode` already defines (flag, persisted declaration, detection, default), calling core's `workspace_mode` resolution with `package="vaultspec-rag"` rather than reimplementing precedence or detection locally. Rag's static `vaultspec-rag.builtin.json` MCP definition is replaced with a tokenized, mode-neutral definition in the same shape core's own builtin already uses, so core's existing sentinel-substitution renderer produces the correct `uv run`- or `uvx`-shaped command for rag without a second renderer. Rag's `server doctor` gains mode-and-floor rows mirroring core's doctor output, driven by the same per-package map entry. Rag's `--upgrade` path gains the same upgrade-time mode inference `install-mode`'s Q6 defines for core, applied to rag's own detection evidence. Rag's `--local-only` flag and its `~/.vaultspec-rag/local-only.json` per-host marker are unchanged; that marker selects a storage backend and remains orthogonal to placement mode, so it is not folded into the shared declaration.

## Rationale

The research establishes that rag already shares core's enrollment machinery byte-for-byte and lacks only the mode axis itself (`2026-07-14-install-parity-research`, parity table), so extending the existing `workspace_mode` module and renderer discipline to a second package is a narrower, lower-risk change than building rag a parallel mechanism; this is the direct application of `install-mode`'s own precedent (a single comparator threaded through every consuming surface) to a second repository rather than a new pattern. The `DEV` mode wins over leaving the two-mode model as-is because the research shows the default-group case is rendering-identical to dependency mode today, so the only missing thing is an honest declared label the doctor and a fresh clone can read, not a new render path; adding it costs one enum member and a doctor label, while not adding it leaves `workspace.json` unable to express what is likely to become the ecosystem's default recommended placement for harness tooling. The per-package map wins over an overrides key because the research names companion-mode divergence, not single-project mode, as the likely common case once both core and rag are provisioned in the same workspace; an overrides map optimizes for the rare case and burdens the common one. Warn-only over hard refusal follows directly from the unanimous ecosystem precedent the research cites (no dev-tool refuses runtime-dependency placement) and from this repository's own dependency-mode self-hosting choice in `install-mode`, which a hard refusal would itself violate.

## Consequences

Good: consumers get one coherent, symmetric mode story across both CLIs, with the common companion-divergence case (one package dependency-mode, the other tool- or dev-mode) representable and diagnosable rather than silently inconsistent. The `DEV` mode closes the gap between what `workspace.json` can declare and what `uv sync` actually does with the default group, without adding a fourth render path. Rag's committed-versus-per-machine state inversion (only a per-host marker, no committed declaration) is fixed by reusing core's existing split rather than inventing a rag-specific one.

Bad: schema v2 is a migration liability on a schema that shipped only one release cycle ago; every early `install-mode` adopter's `workspace.json` needs the legacy-read path exercised correctly, and a bug there breaks the very earliest adopters of the parent decision. The mode surface grows from two choices to three in user-facing flags and doctor output, adding UX surface to document and explain (particularly the DEV-renders-as-DEPENDENCY nuance, which is easy to state incorrectly). Cross-repository release coupling is new: core must release the schema v2 and `DEV` member before rag's floor can move, so a core release that slips blocks rag's parity release, and the two repositories' release cadences become coupled in a way they were not before for this axis.

Neutral: rag's `--local-only` backend marker and its heavyweight torch/model/qdrant provisioning stay entirely orthogonal to placement mode, unaffected by this decision; a workspace can independently choose local-only storage and any of the three placement modes. Existing single-package `workspace.json` files with no companion entry continue to resolve exactly as `install-mode` specified, since the per-package map degrades to the single-entry case with no observable behavior change until a second package is provisioned into the same workspace.
