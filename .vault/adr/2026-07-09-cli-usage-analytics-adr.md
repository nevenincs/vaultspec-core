# `cli-usage-analytics` adr: dev-only transcript analytics module for empirical CLI usage grounding | (**status:** `accepted`)

## Problem Statement

The upcoming MCP-server overhaul must choose which vaultspec-core capabilities to expose as MCP tools, which to consolidate, and which to drop. Today that choice would rest on intuition. Two agent transcript corpora exist on this machine that record real usage over thousands of turns: Claude Code project transcripts (about 5052 JSONL files across 47 project directories, plus subagent logs) and Codex CLI rollout sessions. They carry every vaultspec-core invocation, its outcome, and its approximate token cost, but in two divergent schemas: Claude encodes tool calls as `tool_use` content entries with a separate `tool_result` line carrying only an `is_error` boolean, while Codex encodes `function_call` payloads whose `arguments` field is a JSON string, paired with `function_call_output` records carrying an explicit `Exit code: N` line. Commands are almost never bare invocations - the dominant shape is a `cd` prefix chained with `uv run --no-sync vaultspec-core`, roughly 31% are multi-line, shell `for` loops fire N logical calls from one physical line, and heredocs contain the target string without being calls. No tooling exists to normalize these corpora into a comparable record stream, join observed usage against the declared capability inventory in `.vaultspec/reference/cli.md`, and separate genuine invocation misses from by-design non-zero exits (`vault check` and `plan check` exit 1 on findings; `spec doctor` exits 0/1/2; `migrations status` exits 1 when pending). A dev-only, never-shipped analysis module is needed that produces seven metric families: verb hotspots, command+flag n-grams, feature utilization, `--feature` tag usage, tool-call misses via retry sequencing, dead surface versus declared capability, and token cost per command class.

## Considerations

- The two schemas diverge on every axis that matters: call linkage (`tool_use_id` versus `call_id`), exit-status fidelity (boolean plus text inference versus explicit numeric), and cost attribution (per-assistant-message `usage` versus cumulative per-session `token_count` snapshots requiring deltas).
- Exit-status inference on the Claude side must guard a recurring non-fatal `distutils-precedence.pth` venv traceback in stderr that would otherwise inflate the miss rate.
- The corpora contain personal data: paths, prompts, and raw command bodies. Committed artifacts and generated reports must never carry raw command text; the analysis itself must run only against the operator's local home directories.
- The 30-day activity window (`>= 2026-06-09`) filters on transcript activity timestamps, not file mtime, so every parser must read timestamps before deciding to process a session.
- Streaming is mandatory: the corpus is large, sqlite stores are excluded, and per-line JSONL processing keeps memory flat.
- The declared-capability denominator is machine-generated between `vaultspec:generated` markers in `cli.md` and changes whenever the CLI surface changes; any hand-copied table would rot.
- Test standards forbid mocks, stubs, and skips; fixtures must be real files, and pandas is not a project dependency while Pydantic is.
- The module is a one-purpose dev instrument, not a product surface: repo-root `statistic/` sits outside the hatchling wheel (`src/vaultspec_core` only), so shipping exclusion is structural, not procedural.

## Considered options

**Q1, layout and entrypoint.** Chosen: a flat package `statistic/` with four subpackages - `statistic/parsers` (source adapters), `statistic/normalize` (command tokenization and record construction), `statistic/metrics` (the seven metric families as pure functions over record iterables), `statistic/report` (renderers) - plus `statistic/__main__.py` so `python -m statistic` runs the full pipeline while every layer stays importable as a library for ad-hoc notebook-style exploration. Rejected: a single-file script, because seven metric families over two schemas exceeds what one file can keep testable; rejected CLI-only without library import, because the MCP overhaul work will want to re-query intermediate records interactively.

**Q2, record schema.** Chosen: a single Pydantic model `CallRecord` with fields `source` (literal `claude`/`codex`), `session_id`, `timestamp`, `project` (slug or cwd-derived), `cwd`, `git_branch` (nullable, Claude-only), `verb`, `subcommand` (tuple of path segments, e.g. `("vault","check","links")`), `flags` (canonical name to value mapping with short forms folded into long forms), `feature_tag` (nullable, from `--feature`/`-f`), `command_hash` (SHA-256 of the normalized command, never the raw text), `exit_status` (a small enum: `ok`, `findings`, `error`, `unknown`), `raw_exit_code` (nullable int, Codex only), `token_cost` (nullable), `subagent_role` (nullable), `model`, `cli_version`, and `retry_key` (parentUuid or call_id chain anchor for miss sequencing). Pydantic gives validation at the parse boundary where malformed transcript lines are the norm, and it is already a dependency. Rejected: stdlib dataclasses, which would push per-field coercion and null-handling into every adapter by hand for no dependency savings.

**Q3, parser abstraction.** Chosen: a common `TranscriptSource` protocol with `iter_sessions()` and `iter_calls(session)` implemented by `ClaudeSource` and `CodexSource` adapters; each adapter owns its schema quirks entirely, so the Claude adapter owns `is_error`-plus-text exit inference including the `distutils-precedence.pth` stderr guard, and the Codex adapter owns `Exit code: N` extraction, JSON-string argument decoding, and token-delta computation from cumulative snapshots. Downstream code sees only `CallRecord` iterables. Rejected: two parallel end-to-end pipelines merged at report time, which duplicates the tokenizer and window filter and lets the two sides drift in normalization semantics - the exact defect the module exists to eliminate.

**Q4, tokenizer depth.** Chosen: pragmatic two-stage extraction in `statistic/normalize` - first strip ANSI and CRLF, mask heredoc bodies (between `<<TOKEN` and its terminator) so embedded mentions never count as calls, split on shell connectors (`&&`, `;`, `|`, newline) and unroll single-line `for` loops into N logical segments, then apply `shlex.split` per segment to locate the `vaultspec-core` argv (skipping `cd`, `uv run --no-sync` wrappers, and trailing `2>&1 | head` pipes) and parse verb, subcommand path, and flags positionally. Rejected: a full shell grammar parser (bashlex or a tree-sitter grammar), which adds a dependency and parse-failure modes for marginal accuracy gain, and rejected bare regex-only matching, which cannot handle quoting, multi-line commands, or loop unrolling reliably.

**Q5, declared-capability encoding.** Chosen: parse the command inventory from `.vaultspec/reference/cli.md` between the `vaultspec:generated` markers at analysis time, building the denominator set of verb paths and their declared flags; flag validation is advisory - an observed flag absent from the inventory is reported as a candidate miss, never silently dropped, because the generated reference may lag the installed binary. Rejected: a hand-authored capability table, which rots the moment the CLI surface changes and reintroduces the intuition the module exists to replace.

**Q6, outputs.** Chosen: two artifacts written to `statistic/out/` - `records.jsonl` (the full normalized `CallRecord` stream for re-analysis) and `report.md` (the seven metric families rendered for human reading, containing only aggregates, hashes, and verb paths, never raw command bodies). `statistic/out/` is gitignored as a whole; the code under `statistic/` is committed. Rejected: writing reports into `.vault/` or `docs/`, which would put personal-data-derived output on a committed path and conflate a dev instrument with project documentation.

**Q7, dependencies.** Chosen: stdlib plus Pydantic only - `json`, `pathlib`, `shlex`, `hashlib`, `collections.Counter`, `itertools` cover streaming aggregation comfortably at this scale. Rejected: pandas or polars, which buy columnar convenience the streaming design deliberately avoids, add a dependency the project does not carry, and invite whole-corpus loading.

**Q8, testing.** Chosen: small, hand-redacted fixture transcripts committed under `tests/statistic/fixtures/` - a synthetic Claude project directory and a synthetic Codex session tree, each a few dozen lines exercising every schema edge (multi-line commands, for-loop unrolling, heredoc false positive, `is_error` venv-noise line, by-design exit-1 `vault check`, cumulative token snapshots). Tests point the sources at the fixture roots via explicit path parameters, so they never touch the operator's `~/.claude` or `~/.codex`, and assert exact record counts, hashes, and metric values for determinism. Rejected: fixtures generated at test time from templates, which obscures what the parser actually sees, and rejected testing against live home directories, which is nondeterministic and leaks personal data into assertions.

## Constraints

- Activity window fixed at timestamps `>= 2026-06-09`; filtering on transcript activity timestamps, never file mtime.
- Repo-root `statistic/` only; the hatchling wheel packages `src/vaultspec_core` exclusively, so the module is structurally unshippable and must stay that way.
- Line-by-line streaming; JSONL sources only; the multi-GB sqlite stores are out of scope.
- No raw command bodies, secrets, or personal paths in any committed artifact; generated outputs gitignored.
- By-design non-zero exits (`vault check`/`plan check` findings, `spec doctor` 1/2, `migrations status` pending) map to `exit_status: findings`, never to `error`.
- Project standards apply to the module itself: `ty` type checking, `prek`/`ruff`, Google docstrings with Sphinx refs, zero mocks/stubs/skips.

## Implementation

We will build `statistic/` as a committed dev package with `parsers`, `normalize`, `metrics`, and `report` subpackages and a `__main__` entrypoint. `ClaudeSource` and `CodexSource` implement the `TranscriptSource` protocol, discover session files under their respective home roots (including `subagents/` and `archived_sessions/` via `session_index.jsonl`), apply the 30-day activity filter per line, and emit validated `CallRecord` instances; each adapter owns its linkage model (tool_use_id versus call_id), its exit-status derivation (boolean-plus-text inference with the venv-noise guard versus explicit exit-code extraction), and its cost attribution (per-message usage versus token-snapshot deltas). The normalizer strips ANSI/CRLF, masks heredocs, splits connectors, unrolls loops, and shlex-parses each segment into verb path, canonical flags, feature tag, and a SHA-256 command hash. The metrics layer computes the seven required families as pure functions over the record stream, joining against the capability denominator parsed live from the generated markers in `cli.md`; miss detection sequences records by `retry_key` to distinguish corrected retries from standalone errors. The report layer writes `records.jsonl` and an aggregate-only `report.md` into gitignored `statistic/out/`. Tests run entirely against committed redacted fixtures under `tests/statistic/fixtures/` with exact-value assertions.

## Rationale

The single-protocol adapter design is the load-bearing decision: every downstream metric compares Claude and Codex behavior, so schema divergence must be resolved at exactly one boundary or the comparison is meaningless. Pydantic at that boundary is justified by the data quality of real transcripts, where malformed lines are routine and silent coercion errors would corrupt every aggregate. The pragmatic tokenizer wins over a full shell parser because the corpus documents a small, dominant set of command shapes; handling those shapes explicitly plus shlex covers the observed distribution, while a grammar parser optimizes for shapes that do not occur. Parsing the capability denominator from the generated markers is the only option consistent with the module's purpose - measuring reality against the declared surface requires the declared surface to be read, not remembered. Stdlib-plus-Pydantic keeps the dev instrument installable in the existing environment with zero dependency negotiation, and the streaming design it forces is the same design the corpus size demands anyway. The three-way exit-status enum (`ok`/`findings`/`error`) encodes the by-design exit-code semantics directly into the schema, making it impossible for a metric to conflate a `vault check` finding with an invocation miss.

## Consequences

Good: the MCP overhaul gains an empirical, re-runnable denominator for every capability decision - dead verbs, hot paths, flag n-grams, per-class token cost, and a true miss rate cleansed of by-design exits and venv noise. The `records.jsonl` intermediate lets follow-on questions be answered without re-parsing the corpus. Fixtures document both transcript schemas durably, which has value beyond this module.

Bad: the Claude-side exit status remains an inference, so its error attribution carries irreducible uncertainty relative to Codex's explicit codes; per-call token cost on the Claude side is per-message approximate and on the Codex side delta-derived, so cost comparisons across sources are directional, not exact. The tokenizer's pragmatic depth means pathological shell constructs outside the documented shapes may be miscounted; these will surface as `unknown` exit statuses rather than silent misclassification.

Neutral: the module accrues no maintenance obligation to the shipped product - it can be deleted after the overhaul without trace - but the capability parser does couple it to the `cli.md` marker format, so a marker-format change breaks the denominator loudly at parse time. Analysis validity is bounded to this machine and this 30-day window; conclusions generalize to this operator's usage, not to all users.
