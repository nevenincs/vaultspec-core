# `install-mode` adr: `provisioning is mode-aware and tool-first` | (**status:** `accepted`)

## Problem Statement

Vaultspec-core is development-harness tooling: it governs a project but is not a runtime dependency of it. Provisioning today has exactly one shape and does not know it has made a choice. The scaffolded MCP config hardcodes `uv run python -m vaultspec_core.mcp_server.app`, all four canonical pre-commit hooks build their `entry:` from `CANONICAL_ENTRY_PREFIX = "uv run --no-sync vaultspec-core"`, and the drift detector flags any deviation from that prefix as broken. Every one of these assumes `uv run` resolves inside the target project's own venv, i.e. that `vaultspec-core` sits in the project's dependency set. Install, sync, and doctor never probe `pyproject.toml` and never branch on whether a Python project exists at all; a non-Python repo provisions cleanly and only fails later, opaquely, when a hook or the MCP launch has nothing to resolve `vaultspec_core` from. This is not a bug in one comparator, it is a load-bearing default nobody chose: bundling harness tooling into a project's own dependency set is the wrong default, and provisioning must become explicitly mode-aware to fix it.

## Considerations

- The coupling is total, not partial: MCP config (`src/vaultspec_core/builtins/mcps/vaultspec-core.builtin.json`), all four canonical hooks (`core/commands.py`), and the doctor's own drift signal (`core/diagnosis/collectors.py`, `core/resolver.py`) each independently assume dependency mode. A mode-aware fix must touch all three or reintroduce the same class of surface disagreement the `diagnosis-surface-parity` decision already fixed once for content comparison.
- No committed settings surface exists under `.vaultspec/` today; everything there is framework content, not configuration. `.vaultspec/providers.json` (`ManifestData`) is the only file-backed state, and it is gitignored and per-machine by design, the wrong home for a value contributors must share.
- Version-skew machinery already exists and already assumes same-environment execution: `_resolve_version_warning` compares the running package version against `manifest.vaultspec_version`. Tool mode breaks that assumption outright, since the CLI process and the MCP/hook runtime are no longer guaranteed to be the same install.
- The Windows exe-lock class that originally forced module invocation for the MCP server (commit `18d13bc8`) does not disappear in tool mode, it relocates: the uv tool bin dir copy can be locked by a connected client exactly as the project venv copy could, confirmed upstream as astral-sh/uv issue 11930. Ephemeral `uvx` invocation and module invocation both remain valid mitigations in either mode.
- Ecosystem precedent (pre-commit, Terraform, Cargo, husky, ruff-pre-commit, nx) converges on one shared pattern: a declarative floor constraint committed to the repo, checked at invocation start, hard-refuse with a human-directed remediation. None of them auto-upgrade or persist a separate provisioned-version stamp from the constraint. Vaultspec's nearest in-repo analogue for "warn on drift from what was provisioned" is `BuiltinVersionSignal`, which is complementary to a floor constraint, not a substitute for one.
- This repository is itself dependency-mode and self-hosting; every hook entry here already uses the canonical `uv run --no-sync vaultspec-core` prefix. Dependency mode is not being deprecated, it is being named as one of two explicit, equally first-class choices.

## Considered options

**Q1 - persistence home.** Chosen: a committed shared declaration (a new field on a committed `.vaultspec/` surface, e.g. `.vaultspec/workspace.json`) as the source of truth for the team, with a per-machine echo written into the existing gitignored `providers.json` `ManifestData` for local bookkeeping alongside `installed_at` and `vaultspec_version`. Rejected: manifest-only persistence, because `providers.json` is gitignored and per-machine, so contributors would silently diverge on which mode their own workspace believes it is in. Rejected: env-var-only, because the config system's env-var layer is transient per-shell and leaves no durable record for `doctor` or a fresh clone to read; the ecosystem precedent is unanimous that this class of decision is a committed constraint, not ambient state.

**Q2 - tool-mode MCP launch.** Chosen: module invocation via `uvx --from vaultspec-core python -m vaultspec_core.mcp_server.app`, mirroring the module-invocation mitigation dependency mode already uses, immune to the exe-lock class in both modes, and cache-fast on `uvx` after first resolution. The builtin MCP definition needs either template substitution on the `command`/`args` fields keyed by the persisted mode, or a second mode-variant builtin definition selected at sync time; template substitution is preferable since the definitions are otherwise identical and a second builtin risks the two copies drifting the way the diagnosis surfaces once did. Rejected: bare `vaultspec-mcp` on PATH, because it silently breaks for any contributor who has not separately run `uv tool install`, and no MCP client auto-provisions a missing PATH entry. Rejected: a pinned-exe form as the default, because pinning belongs to the explicit version-constraint mechanism (Q4), not to the baseline launch command.

**Q3 - tool-mode hook entries.** Chosen: `entry: uvx --from vaultspec-core vaultspec-core ...` under `language: system`, preserving the repo's existing `language: system` convention and prek compatibility while resolving self-contained rather than through a project venv that does not exist in tool mode. Rejected: a bare PATH entry, same silent-breakage failure as Q2's rejected option. Rejected: `language: python` with `additional_dependencies: ["vaultspec-core==X.Y.Z"]`, the ruff-pre-commit precedent where pre-commit's own isolated env owns the pin; this is a legitimate alternative but was not chosen as the default because it introduces a second, pre-commit-managed installation of `vaultspec-core` distinct from the one the MCP server and CLI fallback use, which multiplies the version-skew surface Q4 already has to reconcile, whereas `uvx` reuses the same cache-backed resolution path as Q2.

**Q4 - mode-aware diagnosis and version-skew handshake.** Chosen: `CANONICAL_ENTRY_PREFIX` (and the doctor's canonical-entry check that depends on it) becomes mode-parameterized, deriving the expected prefix from the persisted mode rather than a single hardcoded string, so a correctly-provisioned tool-mode workspace is no longer diagnosed as broken. Layer a committed floor constraint (a `minimum_vaultspec_version` alongside the persisted mode, refuse-and-tell at invocation start per the pre-commit/Terraform precedent) on top of the existing `_resolve_version_warning` provision-stamp comparison, which continues to compare the running package version against the manifest's recorded `vaultspec_version` as a softer drift warning. Tool mode changes what that comparison means: dependency mode's implicit assumption that "the running package is the hooks-and-MCP runtime" no longer holds once hooks and the MCP launch resolve through `uvx`'s independently-cached environment, so the floor constraint becomes the hard guarantee and the provision stamp becomes an informational drift signal rather than a correctness guarantee.

**Q5 - provision-time detection and precedence.** Chosen: `--mode` flag (explicit) takes precedence over the persisted mode in the workspace declaration, which takes precedence over detection, which takes precedence over the default. Detection signals, in order: absence of any `pyproject.toml` forces tool mode (nothing exists to resolve a dependency against); presence of `vaultspec-core` in the target's dependencies or dev-dependency group is evidence of deliberate dependency mode. Absent both signals, install defaults to tool mode. Install must refuse coherently when the signals conflict with an explicit request, e.g. `--mode dependency` in a repo with no `pyproject.toml` is a hard refusal with a remediation message, not a silent fallback to tool mode, since silently overriding an explicit flag reintroduces the opaque-failure-later pattern this decision exists to close.

**Q6 - migration.** Chosen: existing workspaces carry no persisted mode. `install --upgrade` infers a mode using the Q5 detection order against the target's current state (an existing `uv run`-shaped canonical hook entry and a `pyproject.toml` dependency listing is treated as dependency-mode evidence) and records the inferred mode into the new persisted declaration on upgrade, rather than requiring every existing workspace to re-run install with an explicit flag. `doctor` gains a mode-mismatch check: artifacts shaped for one mode (a `uv run`-prefixed hook entry, a non-`uvx` MCP command) present in a workspace whose persisted declaration names the other mode are flagged with a fix hint pointing at `install --upgrade` or an explicit `--mode` re-run, keeping this check on the same `apply_file_sync`-routed comparator discipline the `diagnosis-surface-parity` decision established rather than inventing a second one.

## Constraints

- Must not regress the existing Windows exe-lock mitigation in either mode; both the MCP launch and hook entries stay on module/uvx invocation forms that never touch a persistently-open executable.
- Must not introduce a second content or parity comparator; the mode-mismatch doctor check routes through the same canonical-comparator discipline `diagnosis-surface-parity` already established, not a parallel one.
- Must not weaken the `mcp-tool-schema` invariant that `invoke` executes the installed `vaultspec-core` binary and that the gateway catalog is parsed from `cli.md`; mode-awareness changes how that binary is launched and located, not the invariant that the catalog and the binary are the same artifact.
- This repository stays dependency-mode by explicit choice (self-hosting, existing `pyproject.toml` dependency and canonical hook entries), so both modes must remain exercised by the test suite; there is no "one mode is legacy" framing available to narrow test scope.
- `uvx`'s first-run network fetch and cache-prune refetch are outside vaultspec's control; the design cannot assume the launch is always instantaneous.
- Depends on the already-accepted `mcp-tool-schema` and `firmware-mcp-primacy` decisions remaining stable; this decision does not reopen the MCP tool surface or the CLI/MCP wording split, only the transport that provisions and launches the server itself.

## Implementation

Provisioning gains a mode axis threaded through three existing subsystems rather than a new subsystem: the builtin MCP definition, the four canonical hook entries, and the doctor's canonical-entry and version-skew checks all read the persisted mode instead of a hardcoded dependency-mode assumption. A new committed field under `.vaultspec/` records the chosen mode for the team; the gitignored `providers.json` `ManifestData` continues to record local provisioning bookkeeping and gains an echo of the resolved mode plus the floor-constraint version, both derived from the same resolution the CLI performs at provision time. `install` resolves the mode once, at provision time, via the Q5 precedence chain, and every downstream renderer (MCP sync, hook rendering, doctor's expected-content computation) consumes the resolved mode as an input rather than re-deriving it, following the same "one comparator, several views" shape the diagnosis-surface-parity decision established for content parity. `install --upgrade` performs the Q6 inference for workspaces with no recorded mode. `doctor` adds the mode-mismatch signal alongside the existing content-integrity and version-warning signals, feeding the resolver's existing `ResolutionPlan` machinery rather than a bespoke check path.

## Rationale

The research (`2026-07-13-install-mode-research`) establishes two facts that jointly force this shape: every provisioning surface already independently hardcodes the dependency-mode assumption, including the diagnostic that is supposed to catch drift, and no committed settings home exists to hold a shared choice. A design that added a mode flag without fixing all three coupling points (MCP config, hooks, doctor) would repeat the exact single-comparator-bypasses-the-others defect the `diagnosis-surface-parity` decision already diagnosed and fixed once for content hashing; threading the resolved mode through the same rendering path those three surfaces already share is the direct application of that precedent to a second axis of variation. The persistence split (committed declaration plus gitignored echo) mirrors the ecosystem's own split between committed constraint and local tool state (pre-commit's `.pre-commit-config.yaml` versus its local hook cache, Terraform's `required_version` versus its local plugin cache), which is the same shape vaultspec already uses for `BuiltinVersionSignal` versus the manifest. Choosing `uvx` module invocation for both the MCP launch and hook entries over bare-PATH forms is the direct carry-forward of the module-invocation fix that already solved the Windows exe-lock class in dependency mode; the research confirms the lock class relocates rather than disappearing, so the same mitigation shape applies. Defaulting to tool mode, with dependency mode as an explicit, permanently-supported opt-in that this repository itself exercises, is the fixed mandate; the sub-decisions above are the narrowest set of mechanism choices that make that default coherent without reintroducing the opaque runtime failures the research documents.

## Consequences

Good: a non-Python or tooling-only repo can adopt vaultspec without vaultspec-core ever entering its dependency set, closing the failure mode where provisioning silently succeeds and only fails at hook or MCP runtime. Dependency mode remains a fully first-class, tested path, so this repo's own self-hosting is unaffected. The doctor stops flagging correctly-provisioned tool-mode workspaces as drifted. The floor-constraint plus provision-stamp layering gives tool mode an explicit, ecosystem-precedented answer to version skew instead of silently inheriting an assumption that no longer holds.

Bad: version skew becomes a first-class failure mode that did not meaningfully exist before, since tool mode breaks the implicit "running package equals hooks-and-MCP runtime" guarantee dependency mode enjoyed for free; the floor-constraint handshake is new surface to build, document, and keep users from ignoring. `uvx`'s first-run network fetch adds latency to first-time hook execution and MCP launch, and cache pruning (`--refresh`, explicit prune, or an `@latest`-style pin) forces a refetch at an unpredictable time. Both modes must now be exercised by CI and the test suite indefinitely, since this repository staying dependency-mode by choice means tool mode has no dogfood coverage from the project's own workflow and must be tested deliberately. Blast radius is real but bounded: `docs/MCP.md`, the CLI reference, the README, and all four canonical hook definitions need mode-aware rewording or rendering, and the doctor's existing canonical-entry and version-warning checks both need the mode-parameterization from Q4.

Neutral: existing dependency-mode workspaces see no behavior change at their current provisioned state; the migration inference in Q6 only activates on `install --upgrade`. The builtin MCP definition's move from a single hardcoded copy to a mode-templated (or mode-variant) definition is a one-time authoring change with no further ongoing cost once template substitution lands. The `mcp-tool-schema` and `firmware-mcp-primacy` decisions are unaffected in substance; this decision only changes how the MCP server process is located and launched, not what tools it exposes or how firmware names them.
