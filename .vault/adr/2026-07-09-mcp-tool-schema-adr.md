# `mcp-tool-schema` adr: tiered hot-tool surface with a stateless discover/invoke gateway | (**status:** `accepted`)

## Problem Statement

The vaultspec-core MCP server exposes two tools, `find` and `create`, against a CLI of 124 declared verb paths. Agents operating the harness need to orient (project status, feature inventory, in-flight plans, plan grounding, pending steps), check vault health, and create, edit, and batch-mutate vault documents and plan structure - and none of that exists on the MCP surface today. A 1:1 port is off the table: standard MCP clients degrade past roughly 50 advertised tools because every schema is serialized into context on every request, and 124 tools would flood the host before the first user turn. Empirical usage data from the `cli-usage-analytics` audit sharpens the problem: 30 days of real transcripts show usage concentrated on seven verbs (`vault plan step check` at 3.5k invocations, `vault add` 2.6k, `vault plan status` 1.65k, `vault plan step add` 1.5k, `vault feature index` 1.35k, `vault check all` 1.23k, top-level `status`), 38 of 124 verbs never invoked, and a 7.9% miss rate concentrated on exactly the hot verbs - bad flag guesses followed by retry-after-help. The design task is therefore twofold: a small, precisely-schematized first-class tool set that buys down the observed miss rate on the hot path, and a progressive-discovery mechanism that makes the long tail reachable without occupying context. The server runs the FastMCP bundled in the official `mcp` SDK (`mcp>=1.26.0`), not the standalone `fastmcp` package, so every discovery mechanic must be justified against that capability floor.

## Considerations

- The stable protocol revision `2025-06-18` provides tool annotations (`readOnlyHint`, `destructiveHint`, `idempotentHint`, `openWorldHint`), `outputSchema` with `structuredContent`, `resource_link` result content, `tools/list` pagination, and `listChanged` notifications. The `2026-05-21` release candidate moves toward a stateless core (no `initialize` handshake, no session id), full JSON Schema 2020-12 input shapes, and freshness metadata on `tools/list`. Any mechanism that depends on per-session server-side tool-set state swims against the RC current; a stateless request/response discovery mechanism swims with it.
- The SDK-bundled FastMCP reliably offers `@mcp.tool` with `ToolAnnotations`, structured output derived from return types, `Context` logging, and low-level `listChanged` - but not the standalone package's `enable`/`disable`, tag-based set switching, `ToolTransform`, or built-in Tool Search. Dynamic exposure on the current dependency must be hand-rolled.
- Progressive disclosure is the production default in mid-2026 MCP design, with named patterns ranging from meta-tool search gateways through Strict/Sticky/Bounded-Sticky Lazy loading to code-mediated invocation. Independent evidence shows tool description quality materially affects agent efficiency, so the promoted tools' descriptions and schemas are load-bearing.
- The miss-rate finding inverts the usual instinct: the tools agents use most are where they fail most. Schema precision on the hot tools (enums instead of freeform strings, explicit required fields, structured per-item results, actionable errors) is the primary lever, ahead of surface breadth.
- Flag co-occurrence data fixes the canonical signatures: `--feature` with `--related` dominates document creation; `--feature` with `--step` and `--action` with `--scope` dominate plan authoring. Tool signatures should model these shapes directly rather than expose flag bags.
- The current `create` returns `{"success": false}` dicts instead of protocol `isError` results, and `find` inlines document bodies instead of returning `resource_link`s - both are pre-structured-output idioms the redesign must reconcile.
- CLI invariants are absolute: no hand-written frontmatter or filenames, document creation and plan-structure mutation route through the owning verbs' logic, canonical plan identifiers and gap-no-reuse are preserved by that logic and by nothing else. Body prose is the only permitted edit surface.
- The command inventory between the `vaultspec:generated` markers in the shipped CLI reference is machine-generated from the same codebase as the binary, so it is the one catalog source that cannot advertise a verb the installed binary lacks.

## Considered options

**Q1 - surface shape.** Chosen: a tiered surface - nine first-class tools (seven hot-path tools plus a two-tool discovery gateway) with the entire remaining verb surface reachable only through the gateway. Rejected: a curated flat surface of hot tools with the long tail simply dropped, because the user mandate requires the full harness to be operable and the 38-dead-verb finding covers only one operator's 30-day window, not the capability's existence.

**Q2 - discovery mechanism.** Chosen: a hand-rolled meta-tool gateway - `discover` searches a verb catalog and returns full parameter schemas as tool results, `invoke` executes a cataloged verb by path with a validated argument object. This is the Strict Lazy pattern realized as a search gateway: schemas enter context only when fetched, nothing persists server-side, every call is self-contained. Rejected: tag-grouping with `listChanged` set-switching, because the SDK-bundled FastMCP has no enable/disable or transform layer to drive it, host handling of mid-session `listChanged` is uneven, and per-session advertised-set state contradicts the RC's stateless direction. Rejected: code-mediated invocation, as the most invasive pattern with no sandbox story in this harness.

**Q3 - dependency fork.** Chosen: stay on `mcp>=1.26.0` and hand-roll the gateway. The decisive observation is that the gateway does not need the standalone package's headline features at all: long-tail verbs are never registered as MCP tools, so there is nothing to enable, disable, or transform - `invoke` dispatches directly to the owning verb logic. What remains to hand-roll is two meta-tools and a catalog parser, a small and fully-owned surface. Rejected: adding standalone `fastmcp` 2.x/3.x, because it buys built-in Tool Search and set-switching this design deliberately does not use, adds a fast-moving dependency alongside the SDK's own FastMCP with two same-named frameworks importable in one package - a confusion hazard directly adjacent to the settled no-`mcp`-subpackage shadowing rule - and creates a second upstream to track for zero retained capability.

**Q4 - orientation as tools vs resources.** Chosen: tools primary, resources as best-effort read-only mirrors. `status` and `find` are tools because orientation is agent-initiated and parameterized (a feature filter, a plan target, a date window), which is tool semantics; every must-have works on a tools-only host. Vault document bodies and the verb catalog are additionally exposed as addressable resources, and `find` returns `resource_link` entries for document bodies where the host supports them, with the existing `body` inline flag as fallback. Rejected: resources-primary orientation, because resource support across hosts is inconsistent (several clients require manual attachment), which would make the core orientation loop host-dependent.

**Q5 - edit and batch shapes.** Chosen: batch-native single tools. `create` takes a list of document specs (feature, type, title, related, tags, optional seed content - modeling the dominant feature-plus-related shape); items apply sequentially and lifecycle-dependency validation evaluates each item against vault state including earlier items in the same batch, so one call can coherently scaffold research, ADR, and plan for a feature. `edit` takes a list of operations targeting a document by stem or path with an operation enum (`append_section`, `replace_section`, `set_body`) and content, confined to body prose and routed through the set-body-equivalent core logic that owns the `modified` stamp - frontmatter and filenames stay unreachable. Both return structured per-item result arrays; item failures do not abort the batch and are reported per item, with no all-or-nothing transaction since vault documents are independent files and partial success is the honest outcome. Plan structure gets two dedicated tools rather than riding `edit`: `plan_progress` for completion marking and `plan_edit` for step-level authoring, both routed through the `vault plan step` verb logic that guarantees canonical identifiers and gap-no-reuse. Rejected: separate single and batch tool pairs (`create` plus `create_batch`), which doubles the hot-path tool count for identical semantics - a one-element list is the single case. Rejected: freeform body edits via raw file writes, which bypass the owning-verb invariant.

**Q6 - annotations and structured output.** Chosen: `status`, `find`, and `discover` carry `readOnlyHint` and `idempotentHint` true with `openWorldHint` false; `create` is non-read-only, non-destructive, and non-idempotent (repeat creation of an existing file fails, so the current idempotent hint is corrected); `edit` and `plan_edit` are non-idempotent with `destructiveHint` true (`set_body` and `replace_section` overwrite prose); `plan_progress` accepts only explicit `checked`/`unchecked` states - toggle is deliberately excluded from the MCP surface as inherently non-idempotent and retry-hostile - making it idempotent; `check` is read-only without `fix` and loses the hint when `fix` is set, so it is annotated non-read-only, non-destructive, idempotent. All nine tools declare `outputSchema` and return `structuredContent` via FastMCP return-type derivation. Whole-call failures (invalid arguments, unknown verb, unresolvable target) migrate from `{"success": false}` dicts to protocol `isError` results; per-item success fields survive only inside batch result arrays, where a partially-failed batch is a successful call reporting item outcomes. Rejected: keeping the success-dict idiom for uniformity, because it hides failures from host-level error handling and conflicts with the protocol's error taxonomy.

**Q7 - gateway catalog.** Chosen: a catalog generated at server start by parsing the command inventory between the `vaultspec:generated` markers in the shipped CLI reference, minus a small static denylist of out-of-scope verbs (`uninstall`, MCP registry mutation, index hand-authoring). `discover` ranks over verb paths and descriptions; `invoke` validates the verb path and arguments against the same parsed inventory before dispatch, so discovery can never advertise, and invocation can never accept, a verb the installed binary does not declare - reference and binary ship in the same wheel and regenerate together. Rejected: a hand-curated catalog, which rots the moment the CLI surface changes and reintroduces exactly the intuition the analytics work replaced. Rejected: a separate generated JSON catalog artifact, which duplicates the inventory into a second source of truth with its own sync obligation.

**Q8 - installation and versioning.** Chosen: ship through the existing registry unchanged - the same `.vaultspec/mcps/` builtin definition, the same `spec mcps` lifecycle, the same server command - so hosts pick up the new surface on the next `vaultspec-core sync` with zero registry migration. The tool-schema version is the package version: surfaced in the `initialize` implementation info on the stable protocol, echoed in `status` structured output so it survives the stateless RC where `initialize` disappears, and named in the server `instructions` string. Rejected: a dedicated version resource or per-tool version-suffixed titles, which add surface for a fact two existing channels already carry.

## Constraints

- Package stays under the existing MCP server package; no subpackage named `mcp` (would shadow the SDK - settled by the `mcp-consolidation` decision). Registration flows exclusively through the `spec mcps` registry and the builtin JSON definition; no server runtime start/stop management (out of scope by the `mcp-registry` decision).
- Protocol floor is `2025-06-18`; the design must not depend on `2026-05-21` RC-only features, while remaining stateless-compatible and full-JSON-Schema-compatible.
- Dependency floor is `mcp>=1.26.0` with no standalone `fastmcp`.
- CLI invariants are inviolable: no hand-written frontmatter or filenames anywhere on the tool surface; document creation, body edits, plan mutation, and index maintenance route through the owning verb logic; body prose is the only edit surface; `index` documents remain uncreatable and uneditable via MCP.
- The advertised first-class tool count stays in single digits plus gateway; a flat full-coverage surface is prohibited.
- Concurrent-request isolation via the copied-context wrapper is retained on every handler.

## Implementation

The redesigned surface is nine first-class tools, replacing and extending the current pair:

- `status` - read-only project orientation: feature inventory with lifecycle status, in-flight plans with tier and completion, the next open step per plan, and the tool-schema version; parameterizable by feature or plan for the targeted trace including grounding documents.
- `find` - read-only document discovery: filter by feature, type, and date; returns `resource_link` entries for bodies where supported, inline body on request; retains graph-weight feature ranking.
- `create` - batch-native document scaffolding from templates: a list of specs (feature, type, title, related, tags, seed content), sequential application with intra-batch lifecycle-dependency validation, per-item structured results; regenerating the affected feature index is an automatic side effect, absorbing the 1.35k-invocation manual `feature index` call class.
- `edit` - batch-native body-prose editing: operations (`append_section`, `replace_section`, `set_body`) against documents addressed by stem or path, routed through the owning set-body logic, per-item structured results, index refresh as side effect.
- `plan_progress` - mark plan steps `checked` or `unchecked` by canonical step identifier (batch list) against a plan addressed by feature or stem; returns updated completion counts and the next open step; the single highest-frequency operation in the corpus, so its schema is the most tightly typed on the surface.
- `plan_edit` - step-level plan authoring: an operation list (`add`, `insert`, `edit`, `remove`) carrying the action-plus-scope shape the flag data mandates, routed through the plan verb logic that owns canonical identifiers; phase and wave operations are not first-class and live behind the gateway.
- `check` - vault validation and repair: runs the check suite with optional `fix`, returning structured findings.
- `discover` - the gateway search: query the generated verb catalog, receive ranked verb paths with full parameter schemas and descriptions, loaded into context only on demand.
- `invoke` - the gateway executor: run a cataloged verb by path with a JSON argument object, validated against the parsed inventory before dispatch to the owning verb logic; destructive-annotated at the tool level since the long tail includes mutating verbs.

The catalog module parses the generated command-inventory markers at server start, applies the static denylist, and backs both gateway tools from one in-memory structure. Error handling migrates to protocol `isError` for whole-call failures with per-item success arrays inside batch results. All tools declare output schemas and return structured content; all handlers keep the copied-context isolation wrapper. The registry definition, console script, and package location are unchanged.

## Rationale

The tiered shape follows directly from the two empirical facts that frame the problem: usage concentrates on seven verbs, and failures concentrate on the same seven. Spending the entire always-in-context schema budget on precisely those operations - with enums, explicit states instead of toggles, canonical co-occurrence shapes as named parameters, and structured results - attacks the 7.9% miss rate where it actually occurs, while the gateway keeps the other hundred-odd verbs one search away at near-zero standing context cost. The stateless discover/invoke mechanism is chosen over set-switching not merely because the SDK-bundled FastMCP lacks the switching primitives, but because a request-scoped gateway is the only pattern that is simultaneously implementable on the current dependency, indifferent to host `listChanged` quality, and aligned with the protocol's stateless trajectory - which is also why staying on `mcp>=1.26.0` costs nothing: the standalone package's dynamic-exposure machinery solves a problem this architecture does not have, since long-tail verbs are never MCP tools in the first place. Routing every mutation through owning verb logic is non-negotiable inheritance, and it is what makes batch semantics safe: sequential per-item application through validated verbs yields honest partial results without inventing a transaction layer over independent files. The marker-parsed catalog closes the last integrity gap - discovery, invocation, and the binary derive from one generated inventory, so the gateway is structurally incapable of advertising fiction.

## Consequences

Good: agents gain the full orientation loop (`status`, `find`), health checking (`check`), and the complete create/edit/plan mutation surface the harness workflow requires, in nine tools instead of 124; the hot path gets the schema precision the miss-rate data demands; the long tail stays reachable without context cost; the design survives the stateless protocol transition unmodified; installation is a no-op for existing projects; folding index regeneration into mutation side effects deletes an entire hot manual call class.

Bad: gateway verbs cost an extra `discover` round-trip before first use, and their schemas are only as good as the generated inventory's descriptions - description quality in the CLI reference becomes load-bearing for agent success on the long tail. `invoke` is a single broad tool whose destructive annotation forces conservative host confirmation on every long-tail call, including read-only ones. Hand-rolling the gateway means owning its search ranking and argument validation without upstream support. Batch partial success shifts result-inspection burden onto the calling agent.

Neutral: the two existing tools change contract (`create` gains batch shape and loses its incorrect idempotent hint; error signaling moves to `isError`), which is a breaking schema change absorbed at the next sync rather than a compatibility layer. The static denylist is a small curated list that must be revisited if out-of-scope verb classes change. Resources ship as best-effort mirrors and may go unused on hosts without resource support, which costs nothing.
