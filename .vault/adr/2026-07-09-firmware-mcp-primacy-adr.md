# `firmware-mcp-primacy` adr: per-surface MCP primacy - capability-worded firmware, MCP tools primary at the orchestrator, CLI mechanism retained in personas | (**status:** `accepted`)

## Problem Statement

The always-preloaded firmware under `src/vaultspec_core/builtins/` mandates CLI invocation throughout: `rules/vaultspec-cli.builtin.md` is a ~117-line command catalog culminating in literal `uv run --no-sync vaultspec-core <cmd>` runtime syntax, `system/03-vaultspec.md` carries a MUST-level `vaultspec-core vault plan ...` mandate, and the write-capable agent personas each embed dedicated CLI usage mandates enumerating verb-by-verb invocations. A verified nine-tool MCP server now exists (seven hot-path tools plus a `discover`/`invoke` gateway, per the accepted tool-schema decision) that covers the highest-frequency operations with precisely-typed schemas. Preloading CLI invocation syntax into every session is now a double context spend: the firmware pays for command enumeration the MCP schemas already carry, and agents in MCP-connected sessions are steered toward the miss-prone freeform-flag transport the hot tools were built to replace. Yet the firmware installs into projects where the MCP may not be connected, no runtime probe can tell an agent whether it is, no persona's `tools:` allowlist contains an MCP tool, and several mandated operations have no first-class MCP tool at all. The decision is how far to push MCP primacy, on which firmware surfaces, and with what exact wording architecture.

## Considerations

- The firmware has three standing-context classes with different audiences: always-on rules and the assembled system prompt address the orchestrating session, which may hold the MCP tools; the ten personas address dispatched subagents, which per their own `tools:` allowlists (`Glob, Grep, Read, Write, Edit, Bash` variants, no MCP tool anywhere) reach vaultspec exclusively through `Bash`. Whether a harness propagates MCP tools into subagents is unproven either way; the allowlists make the current answer "no" regardless.
- There is no live-connection signal. `spec mcps status` reports registration-on-disk, not host attachment, and is itself a CLI call. Graceful degradation must therefore be achieved by wording that is true in both worlds, not by conditional logic. The `vaultspec-discovery.builtin.md` trailing fallback clause ("Where `vaultspec-rag` is not installed, the `vaultspec-core` discovery verbs and grep carry the same sequence") is the proven single-clause pattern.
- MCP coverage is asymmetric. Hot-tool equivalents exist for `status`, `vault list`, `vault add`, body edits, `plan step check/uncheck`, `plan step add/insert/edit/remove`, and `vault check all` (`status`, `find`, `create`, `edit`, `plan_progress`, `plan_edit`, `check`). Gateway-only: `sync`, `spec <resource> sync`, and the plan verbs `tier promote/demote`, `wave`, `phase`, `epic intent`. No MCP path at all (denylisted): `vault feature index` (regenerated instead as a `create`/`edit` side effect), `spec mcps add/remove/sync`, `uninstall`. Any "the MCP replaces the CLI" claim is factually wrong for these.
- Both transports terminate in the same owning verb logic; the invariant worth mandating is "route through the owning verbs, never hand-edit", which is transport-neutral. The transport choice is an efficiency preference layered on top of that invariant, not the invariant itself.
- `reference/cli.md` is read-on-demand, never assembled into provider context, so its size costs no standing context; but its `vaultspec:generated` command-inventory marker block is the MCP gateway's authoritative verb catalog, guarded by a drift test. Only prose outside the markers is touchable, and that prose is emitted by the reference generator, so it must change at the generator source.
- A reword is a `sync`-propagated content change with firmware-wide blast radius (every managed project on next sync or install), dogfooded first in this repo's own synced `.claude/rules/` copies. It is not a `migrations`-class change.
- The server stamps its tool-schema version (package version) in its instructions and in `status` output; a firmware naming specific tools is implicitly coupled to a server version that has them.

## Considered options

**Q1 - overall posture.** Chosen: a per-surface split. Always-on rules and the system prompt go MCP-first-with-CLI-fallback: every hot-path mandate names the capability and the primary MCP tool, with the CLI named once as the disconnected-session fallback. The ten personas go capability-first with the CLI retained as the named mechanism: mandates lead with intent and the owning-verb invariant, and the CLI verb remains the stated execution path, because the personas' allowlists make it the only path they have. The read-on-demand reference becomes the explicit fallback-and-gateway catalog. Rejected: uniform MCP-first-with-fallback across all surfaces, because it instructs subagents to prefer tools their allowlists exclude, guaranteeing a dead-first-attempt on every persona dispatch. Rejected: minimal capability-only mention (name intent, let the harness route), because the tool-schema decision's own evidence shows naming the precise tool is load-bearing - the observed miss class was bad transport guesses, and vague wording reintroduces guessing on both transports.

**Q2 - availability detection.** Chosen: none. The firmware relies purely on graceful-degradation wording; no probe, no conditional instructions, no "check `spec mcps status` first" step. Rejected: instructing agents to probe registration via `spec mcps status`, because it reports disk state rather than host attachment, costs a CLI round-trip to decide whether to avoid CLI round-trips, and is circular with the premise. The host either surfaces the tools or it does not; wording that is correct in both worlds needs no detection.

**Q3 - the ten personas.** Chosen: reword to capability-and-owning-verb framing while keeping `Bash` plus `vaultspec-core` as the stated mechanism and leaving every `tools:` allowlist byte-identical. The persona mandates lose duplicated verb enumeration where the intent carries it, but each mutation mandate still names its exact CLI verb once, because that verb is the persona's only reachable transport. One new shared sentence in `system/03-vaultspec.md` makes the split explicit: dispatched personas operate vaultspec through the CLI; MCP tools are not assumed inside subagents. Rejected: adding MCP tools to persona allowlists, because subagent MCP inheritance is an open harness question with no firmware evidence - if inheritance fails, a persona scoped to MCP tools is stranded with no vaultspec access at all, the worst failure mode available; this is revisitable as a follow-on once inheritance is empirically verified, and this record should be superseded on that point then. Rejected: leaving persona wording untouched, because the executor and writer mandates currently interleave transport syntax with the invariant, and separating "route through the owning verb" (permanent) from "invoke it as `vaultspec-core ...`" (transport) is exactly the restructuring that makes a future allowlist change a one-line edit.

**Q4 - the canonical wording pattern.** Chosen: capability first, primary MCP tool named inline, one trailing CLI-fallback clause per rule file or section rather than per sentence. The canonical shape: state the capability as the mandate, name the MCP tool as the primary path in the same sentence, and let a single file-level clause - modeled on the discovery rule's - carry the fallback: "Where the vaultspec MCP server is not connected, the `vaultspec-core` CLI verbs carry every operation; the bundled CLI reference is the catalog." Concrete before/after on the system prompt's strongest mandate. Before: "Writers and executors MUST use the `vaultspec-core vault plan ...` CLI verbs (`step add/insert/move/remove/check/uncheck/toggle/edit`, `phase`/`wave` equivalents, `epic intent`, `tier promote/demote`) for every identifier-affecting change rather than hand-editing the markdown body." After: "Every identifier-affecting plan change MUST route through the owning plan verbs, never hand-edits: mark Step completion with the `plan_progress` tool and author Step rows with the `plan_edit` tool; structural changes above Step level (`phase`, `wave`, `epic intent`, `tier promote/demote`) and any session without the MCP server use the `vaultspec-core vault plan ...` CLI verbs, which guarantee the same canonical-identifier preservation and gap-no-reuse." The after-form is shorter, true whether or not the MCP is connected, honest about the above-Step gap, and keeps the invariant in the MUST position. Rejected: per-sentence dual wording ("use tool X or command Y" on every line), because it doubles token cost precisely where the reword is supposed to save it. Rejected: fallback-free MCP wording, unsafe per Q1.

**Q5 - fate of the CLI rule and the CLI reference.** Chosen: rewrite `rules/vaultspec-cli.builtin.md` in place, same filename, roughly halved. The `## Mandate` becomes transport-neutral ("all vault reads, mutations, audits, and repairs route through `vaultspec-core` owning-verb logic; the MCP tools are the primary transport where connected, the CLI otherwise"). The ~20-command `## Commands` catalog collapses to three short groups: the nine MCP tools mapped to their capabilities; the honest CLI-or-gateway list from Q6; and a pointer stating the `discover` tool and the bundled CLI reference are the catalogs for everything else. The `## Runtime` block shrinks to the two-line CLI-fallback essentials (`uv run --no-sync` in uv environments, `--target`, `--json`, `--help`) under an explicit "CLI fallback" heading. `reference/cli.md` stays: the generated marker block is untouched (it is the gateway's verb catalog, drift-tested, generator-owned), and the human-facing header prose gains one paragraph - edited at its source in the reference generator, not in the emitted file - naming its two roles: the CLI-fallback lookup and the MCP gateway's verb-existence source. No trim beyond that; the file is read-on-demand and costs no standing context. Rejected: deleting or renaming the rule file, which churns the synced-file inventory and the CLAUDE.md rule block across every managed project for zero semantic gain. Rejected: trimming the reference's inventory, which would break the gateway catalog and the drift test.

**Q6 - operations with no first-class MCP tool.** Chosen: a three-band honesty scheme, stated once in the rewritten CLI rule and echoed where each operation is mandated. Band one, hot seven: worded MCP-tool-primary per Q4. Band two, gateway-only (`sync`, `spec <resource> sync`, plan `tier`/`wave`/`phase`/`epic intent`): worded CLI-first with a single mention that the `discover`/`invoke` gateway also reaches them - CLI-first because `invoke`'s destructive annotation forces host confirmation on every call, making the CLI the better default even in connected sessions. Band three, denylisted (`vault feature index`, `spec mcps` mutation, `uninstall`): worded CLI-only, explicitly. The feature-index instruction in `rules/vaultspec.builtin.md` and `system/03-vaultspec.md` is reworded to the new truth: the index regenerates automatically as a side effect of the `create` and `edit` tools; regenerate manually with `vaultspec-core vault feature index` when working through the CLI. Rejected: blanket "the MCP covers the CLI" phrasing, factually false for band three and misleading for band two. Rejected: leaving the old unconditional "run `vaultspec-core vault feature index`" mandate, which instructs MCP-path agents to perform a call class the tool surface deliberately absorbed.

**Q7 - migration and versioning.** Chosen: ship as an ordinary content change to the `.vaultspec/` sources, propagated by `vaultspec-core sync` and the next `install`; no schema migration, no version gating in the wording. Skew is handled by the same degradation clause that handles disconnection: the firmware names tools by capability, and the Q4 fallback clause covers any session where a named tool is absent - whether because the server is not connected or because an older server predates it. The server already stamps its tool-schema version in `status` output and its instructions string; the firmware does not repeat it. Rejected: version-conditional wording ("on servers >= X use tool Y"), because the firmware has no conditional evaluation and version-pinned prose rots on every release. Rejected: a `migrations`-class change, wrong tool - nothing about on-disk schema changes.

## Constraints

- The accepted tool-schema decision is inviolate: nine first-class tools, gateway for the long tail, all mutation through owning verb logic, `index` documents uncreatable and uneditable via MCP. This record changes wording, not the tool surface.
- The `vaultspec:generated` command-inventory block in the CLI reference must remain byte-stable against the generator; only prose outside the markers changes, and only via the generator source so regeneration preserves it. The drift test must stay green.
- Persona `tools:` allowlists do not change in this decision.
- No firmware surface may contain conditional logic on MCP availability; correctness in both worlds is achieved by wording alone.
- Firmware wording must never claim MCP coverage for the denylisted verbs or imply hot-tool status for gateway-only verbs.

## Implementation

Files that change, all under `src/vaultspec_core/builtins/` unless noted, propagated by `vaultspec-core sync` and dogfooded in this repo's synced `.claude/rules/` copies:

- `rules/vaultspec-cli.builtin.md` - full rewrite in place per Q5: transport-neutral mandate, nine-tool capability map, the Q6 three-band CLI-or-gateway list, two-line CLI-fallback runtime block, single file-level fallback clause, retained "Allowed manual edits" and orientation sections reworded to name `status` and `find` as primary. Target roughly half the current length.
- `rules/vaultspec.builtin.md` - three surgical touches: the plan-authority sentence adopts the Q4 after-form's short shape; both `vault feature index` mentions reword to index-as-side-effect with the CLI verb as the manual path; everything else (taxonomy, hierarchy, placeholders) is untouched.
- `rules/vaultspec-discovery.builtin.md` - unchanged; it is already the fallback-clause template this decision generalizes.
- `system/03-vaultspec.md` - the orient-first paragraph names the `status` tool as the primary orientation path with `vaultspec-core status` as the fallback; the feature-index sentence rewords per Q6; the MUST plan-verbs paragraph is replaced with the Q4 after-form verbatim; the Agents section gains the one-sentence subagent clause from Q3 (personas operate vaultspec through the CLI; MCP tools are not assumed inside dispatched subagents).
- `system/01-core.md`, `system/02-operations.md`, `system/90-custom.md` - unchanged.
- `agents/vaultspec-writer.md` - the `## CLI usage mandate` section is retitled to an owning-verbs mandate: invariant first (structural manipulation routes through the plan verbs, never hand-edits), then the CLI verbs named once as this persona's execution path, with the verb enumeration compressed to the verb-group list rather than one sentence per verb. `tools:` unchanged.
- `agents/vaultspec-standard-executor.md`, `agents/vaultspec-low-executor.md`, `agents/vaultspec-high-executor.md` - the Scaffold and step-state mandates keep their exact CLI verbs (`vault add exec ...`, `vault plan step check/uncheck`) but lead with the capability sentence and drop `toggle` from the recommended set, matching the tool surface's explicit-state stance. `tools:` unchanged.
- `agents/vaultspec-docs-curator.md` - same capability-first pass over its seven CLI references; verbs retained.
- The five remaining personas (researchers, auditors, reviewer, coordinator) - unchanged; their CLI footprint is already minimal and discovery-worded.
- `mcps/vaultspec-core.builtin.json` - unchanged.
- `src/vaultspec_core/cli/reference_gen.py` (generator source, not a builtin) - the emitted header prose gains the dual-role paragraph from Q5; the marker block emission is untouched; regenerate via `spec reference generate` and confirm `test_cli_reference_drift` passes.

Rollout: edit builtin sources, run `vaultspec-core sync` here, verify `vault check all` and the full test gate, and ship in the next release; consuming projects absorb it on their next `sync` or `install` with no migration step. A follow-on decision point is registered, not implemented: if subagent MCP inheritance is ever empirically confirmed, supersede this record's Q3 stance and re-scope persona allowlists then.

## Rationale

The split posture follows from the firmware's own frontmatter. The rules and system prompt address a session that may hold nine precisely-schematized tools built specifically to buy down the observed CLI miss rate, so naming those tools as primary there converts standing context from invocation syntax into intent plus a tool name - cheaper and less miss-prone. The personas address subagents whose allowlists provably exclude those tools, so MCP-first wording there would be an instruction to fail; keeping the CLI as their named mechanism is not conservatism but the only factually correct wording. Detection is refused because no honest signal exists: the only pattern that is true in every deployment is wording that states capability, primary tool, and one fallback clause, which the discovery rule has already proven at scale. The three-band honesty scheme exists because the MCP surface was deliberately tiered; firmware that flattens that tiering into "use MCP" would send agents at denylisted verbs the gateway will refuse. Keeping the CLI rule's filename and the reference's inventory intact confines the blast radius to prose, which ordinary `sync` propagates and the drift test polices - the cheapest possible shipping shape for the widest-reaching class of change the framework makes.

## Consequences

Good: standing context in MCP-connected sessions steers every hot-path operation to the tool built for it, and the always-on CLI rule roughly halves; the owning-verb invariant is finally stated transport-neutrally, so future transport changes (including persona allowlist expansion) become wording-local edits; disconnected installs lose nothing, since the CLI remains fully documented as the fallback with its catalog one read away; the feature-index mandate stops instructing a manual call class the tool surface absorbed.

Bad: dual-world wording is inherently longer than single-world wording would be in either pure deployment; sessions without the MCP carry a few sentences of tool-naming they cannot use, and MCP-connected sessions carry the fallback clause they never need. Personas remain on the slower, miss-prone CLI transport until the inheritance question is settled, so the subagent hot path sees none of this decision's efficiency gain. Every managed project's provider rules change on next sync, which will surface as diff noise in downstream repos that commit their provider directories.

Neutral: the CLI reference's generated inventory, the gateway catalog, and the drift test are untouched, so the MCP server behavior is byte-identical before and after; the reword is coupled to server versions only through the same fallback clause that covers disconnection, so no version bookkeeping enters the firmware; the Q3 stance is explicitly provisional pending harness evidence and is expected to be superseded rather than edited.
